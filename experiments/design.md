# 实验设计文档

> 角色：科学研究员（设计）+ 架构师（实现）
> 目标：用工程化方法验证 Hermes 三个核心记忆机制是否真的可行

---

## 实验总则

### 验证原则

1. **不信任设计，只信任数据**。Hermes 说它的机制能让 Agent 越用越好——我们要用实验数据证明或否定这个说法。
2. **每个实验必须有对照组**。没有对照的实验结果没有意义。
3. **指标可量化**。所有结论必须来自数字，不是主观感受。
4. **可复现**。实验数据、参数、代码全部留存，任何人在任何时候可以重跑。

### 共享基础设施

- **LLM 调用**：使用智谱 API（Anthropic 兼容接口），记录每次调用的 token 消耗和延迟
- **Embedding**：使用智谱 embedding API 生成向量
- **数据持久化**：SQLite（轻量、可审计、可复现）
- **结果格式**：每个实验输出 JSON + CSV，便于后续分析

## 实验 1：生命周期状态机

### 研究员视角

**假设**：基于活动时间的状态机（active→stale→archived + reactivation）比简单时间过期更准确地管理知识库。

**零假设**：状态机与简单过期在知识库健康度上无显著差异。

**自变量**：状态管理策略（Hermes 状态机 vs 简单过期）

**因变量**：
| 指标 | 定义 | 期望 |
|------|------|------|
| 状态转换准确率 | 被标记 stale/archived 的记录中，确实应该被标记的比例 | >90% |
| 误杀率 | 被标记 stale 但随后 7 天内被重新引用的比例 | <10% |
| 再激活召回率 | 实际应被重新激活的 stale 记录中，被成功激活的比例 | >80% |
| 知识库活跃率 | active 记录数 / 总记录数 | 保持在 40-70% |

**对照组**：
- **对照组 A（简单过期）**：超过 N 天的记录直接标记为 expired，无 reactivation
- **对照组 B（无管理）**：所有记录永远 active

**实验数据**：生成 1000 条模拟决策记录，时间跨度 180 天，引用频率服从幂律分布（少数记录被频繁引用，多数记录很少被引用）。

**实验步骤**：
1. 生成模拟数据集（含创建时间、最后引用时间、引用频率）
2. 实现 Hermes 状态机逻辑（30天 stale，90天 archived，引用后 reactivation）
3. 实现对照 A（简单过期，30天直接 expired）
4. 模拟 180 天的知识库演变，每天跑一次状态检查
5. 在第 30/60/90/120/150/180 天采样快照
6. 对比三组的指标

### 架构师视角

**依赖**：
- 纯 Python，无外部 API 调用
- SQLite 存储状态
- matplotlib 绘图

**数据生成器**：
```python
# 幂律分布引用频率
# 每条记录：{id, created_at, last_referenced_at, reference_count, content_embedding}
# 引用事件模拟：高引用记录每 1-3 天被引用一次，低引用记录每 30-60 天一次
```

**输出**：
- `results/exp1_timeseries.csv`：每天的指标数据
- `results/exp1_summary.json`：最终对比结果

## 实验 2：知识合并（Consolidation）

### 研究员视角

**假设**：LLM 驱动的 umbrella-building 合并比纯向量相似度合并产生更准确的知识归并。

**零假设**：LLM 合并与向量相似度合并在合并质量上无显著差异。

**自变量**：合并策略（LLM umbrella-building vs 纯 embedding 阈值 vs 人工标注基线）

**因变量**：
| 指标 | 定义 | 期望 |
|------|------|------|
| 合并精确率 | LLM 建议合并的配对中，确实应该合并的比例 | >85% |
| 合并召回率 | 应该合并的配对中，被 LLM 发现的比例 | >70% |
| 误合并率 | 不应合并但被合并的比例 | <15% |
| LLM 调用成本 | 每次合并判断的 token 消耗 | <2000 tokens/judgment |
| 延迟 | 单次合并判断的端到端延迟 | <3s |

**对照组**：
- **对照组 A（纯 embedding）**：余弦相似度 > 0.85 直接合并，不经过 LLM
- **对照组 B（人工标注）**：作为 ground truth，准备 50 对记录的人工标注（该合并/不该合并）

**实验数据**：
- 从 Hermes 技能库模式生成 200 条模拟"技能/知识"记录
- 按主题聚类，确保存在：
  - 明确应合并的对（如"Redis 缓存" + "Redis 消息队列"）
  - 明确不应合并的对（如"Redis 缓存" + "MySQL 索引优化"）
  - 模糊边界对（如"Redis 缓存" + "Memcached 缓存"）

**实验步骤**：
1. 生成 200 条模拟记录，人工标注 50 对作为 ground truth
2. 对所有 200×199/2 = 19900 对计算 embedding 相似度
3. 筛选相似度 > 0.6 的对（减少 LLM 调用量）
4. 用 LLM 对筛选后的对做合并判断（结构化 YAML 输出）
5. 对比 LLM 判断 vs 纯相似度 vs 人工标注

### 架构师视角

**依赖**：
- 智谱 embedding API（向量化）
- 智谱 chat API（LLM 合并判断）
- numpy（相似度计算）

**关键实现**：
```python
# LLM prompt：给定两条记录，判断是否应合并为 umbrella，输出结构化 YAML
# 需要验证：YAML 解析成功率、格式合规率
```

**输出**：
- `results/exp2_pairs.csv`：每对的标注结果
- `results/exp2_metrics.json`：精确率/召回率/误合并率

## 实验 3：上下文预取（Prefetch）

### 研究员视角

**假设**：prefetch 机制（按需从记忆库检索相关上下文）比全量注入或无上下文产生更高质量的 AI 响应。

**零假设**：prefetch 与无上下文在响应质量上无显著差异。

**自变量**：上下文策略（prefetch top-K vs 全量注入 vs 无上下文）

**因变量**：
| 指标 | 定义 | 期望 |
|------|------|------|
| 响应相关性 | AI 回答与问题的相关程度（1-5 分，人工评估） | prefetch > 全量 > 无上下文 |
| 决策关联率 | AI 主动引用历史决策的比例 | prefetch > 30% |
| Token 效率 | 有效信息 / 总注入 token | prefetch 效率最高 |
| 延迟开销 | prefetch 检索的额外延迟 | <200ms |

**对照组**：
- **对照组 A（无上下文）**：纯 system prompt + 用户问题，不注入任何历史
- **对照组 B（全量注入）**：把最近 50 条决策全部塞入 prompt
- **实验组（prefetch）**：根据当前问题 embedding，检索 top-5 相关决策注入

**实验数据**：
- 准备 30 个模拟决策场景（每个场景包含一个问题和已有的 100 条历史决策）
- 每个场景的 100 条历史决策中，人工标注 3-5 条为"相关"

**实验步骤**：
1. 为 30 个场景生成历史决策库
2. 对每个场景，分别用三种策略生成 AI 响应
3. 人工盲评响应质量（5 分制）
4. 自动评估：AI 响应中引用了标注的"相关决策"的比例

### 架构师视角

**依赖**：
- 智谱 embedding API + chat API
- 余弦相似度检索（numpy）

**关键实现**：
```python
# prefetch：query_embedding → cosine_sim against all decisions → top-K
# 全量注入：最近 50 条按时间排序
# 无上下文：仅 system prompt + query
# 评估：LLM-as-judge 辅助 + 人工标注
```

## 实验 4：上下文压缩（3-Phase Compression）

### 研究员视角

**假设**：3-phase 压缩（prune → protect → summarize）比简单截断在信息保留和压缩比上更优。

**零假设**：3-phase 压缩与简单截断在信息保留上无显著差异。

**自变量**：压缩策略（3-phase vs 简单截断 vs 仅摘要）

**因变量**：
| 指标 | 定义 | 期望 |
|------|------|------|
| 压缩比 | 压缩后 token / 压缩前 token | <40% |
| 关键事实保留率 | 人工标注的关键事实在压缩后仍存在的比例 | >80% |
| 决策链完整性 | 压缩后的摘要是否保留完整决策链（背景→决策→理由） | >70% |
| 压缩延迟 | 端到端压缩时间 | <5s |
| LLM 调用成本 | 压缩一次的 token 消耗 | 可量化 |

**对照组**：
- **对照组 A（简单截断）**：超过 token 预算时，从中间截断，保留头部和尾部
- **对照组 B（仅摘要）**：不分 phase，直接让 LLM 做一次全文摘要

**实验数据**：
- 构造 10 组模拟的长对话上下文（每组 20000-50000 tokens）
- 每组人工标注 15 个关键事实

**实验步骤**：
1. 生成 10 组长上下文数据
2. 每组标注 15 个关键事实
3. 对每组分别用三种策略压缩到 8000 tokens
4. 自动检查关键事实是否保留在压缩结果中
5. 人工评估决策链完整性

### 架构师视角

**依赖**：
- 智谱 chat API（摘要 LLM 调用）
- tiktoken 或等价 tokenizer（token 计数）

**3-phase 实现**：
```python
# Phase 1 (Prune): 工具输出替换为 1-line summary
# Phase 2 (Protect): 保护 system prompt + 最近 20K tokens
# Phase 3 (Summarize): LLM 结构化摘要（Active Task, Goal, Key Decisions...）
# 参数: threshold_percent=50%, summary_target_ratio=20%
```

## 风险

1. **LLM 输出不稳定**：同一 prompt 多次调用结果可能不同。对策：每个实验重复 3 次取平均。
2. **模拟数据偏差**：人工生成的测试数据可能不代表真实场景。对策：标注明确，后续可用真实数据验证。
3. **API 成本**：4 个实验预计消耗约 500K-1M tokens。对策：先小规模跑通，再扩大数据量。
